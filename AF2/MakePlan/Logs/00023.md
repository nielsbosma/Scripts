Do an extensive analysis of all runtime errors that can happen in the framework. Is there any good candidate to make analysers in Ivy.Analyser to get compile time error instead of runtime error.

## Outcome

Conducted extensive analysis of 200+ runtime error locations across the Ivy-Framework codebase. Identified 14 categories of runtime errors and created 5 implementation plans for the highest-impact analyzer candidates.

### Plans Created

| ID | Title | Impact |
|----|-------|--------|
| 396 | No-Children Widget Analyzer (IVYCHILD001/002) | HIGH - 23 widgets affected, very common mistake |
| 397 | Hook Results in Fields Analyzer (IVYHOOK006) | HIGH - Silent bugs, extremely hard to debug |
| 398 | IStateless View Hook Analyzer (IVYHOOK007) | MEDIUM - Clear runtime error, easy to detect |
| 399 | DropDownMenu Child Type Analyzer (IVYCHILD003) | MEDIUM - Type-restricted children |
| 400 | Build Return Type Analyzer (IVYVIEW001) | MEDIUM - Common beginner mistake |

### Runtime Error Categories Found (Not Planned)

Additional runtime errors that were analyzed but NOT selected for analyzers (either infeasible or low ROI):
- Configuration validation failures (8+ auth providers) - Not feasible as compile-time check
- Reflection-based type lookups - Would need source generators, not analyzers
- LINQ `.First()`/`.Single()` without checks - Too general, not Ivy-specific
- JSON parsing without validation - Runtime data, can't check at compile time
- OAuth flow state machine violations - Complex state, not statically analyzable
- Widget state type mismatches - Already caught by generic constraints
- URL/string validation - Runtime data
